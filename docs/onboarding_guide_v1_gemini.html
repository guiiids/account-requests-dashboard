iLab Account Requests Dashboard

Onboarding Guide for Support Staff & Administrators

How the System Works

The iLab Account Requests Dashboard is an internal tool that replaces manual email tracking for iLab signup requests. Here's the big picture:

Email arrives iLab signup request
Dashboard ingests Creates ticket
Agent picks up Ticket from queue
Agent works ticket Notes, emails, status
Ticket closed When resolved

Key Concepts

  • Requests arrive automatically from a shared mailbox via Power Automate.
  • Smart Threading groups email replies with the original request using Outlook Conversation IDs.
  • Tabbed Interface lets you work multiple requests simultaneously — like browser tabs.
  • Every action is audited — status changes, emails sent, notes added, and logins.
1

Logging In

Navigate to the dashboard URL and you'll see the login screen.

Field What to enter
Email Address Your @agilent.com email
Password Your assigned password
Remember me Check to stay signed in across browser sessions
First Login: If this is your first time logging in, you'll be asked to change your password immediately. Choose something at least 8 characters long.
Lockout Protection: After 5 failed login attempts, your account is temporarily locked for 15 minutes. Wait and try again or contact an admin.
2

The Dashboard (Home Base)

After logging in, you land on the Dashboard — your command center for all account requests. The interface has three main zones:

The Sidebar (Left)

Your navigation hub. It stays visible on every page.

  • Overview: Main dashboard with request queue (you are here)
  • Import Data: Manually paste email content to create a request
  • Audit Log: View a read-only record of all agent actions
  • Users: Manage staff accounts (Admin only)

The Request Queue (Table)

The main table listing all requests. Each row shows:

  • Reference: Unique ticket ID (e.g., ACCT-001)
  • Subject: Email subject line with requester's name and organization
  • Status: New, Waiting, or Closed — color-coded badge
  • Owner: Which agent is assigned (or — if unassigned)

Opening a Request (Tabbed Detail View)

Click any row in the request table to open it in a new tab — just like browser tabs, but within the dashboard itself. The detail view is split into two columns:

Left Column — Activity Stream

The conversation history, shown newest-first. Each message is a card showing:

  • Avatar with initials (blue = Agent, gray = Requester, light = System)
  • Role badge — Support, Requester, or System
  • Timestamp — auto-converted to local timezone
  • Message body — full email or note content

Right Column — Metadata Sidebar

Header Ticket ID and subject line
Details Request type, creation date, iLab link
Status & Owner Dropdown selectors
Requester Name, email, organization, lab
Quick Actions Reply via Email & Close/Reopen buttons
5

Working a Ticket

Once you've opened a request, here's the typical workflow:

1. Assign to yourself Set assignee in sidebar
2. Change status → "Waiting - for Support"
3. Add internal note Document the resolution

Changing Status

Click the Status dropdown in the sidebar. Options:

Status Badge When to use
New New - Open Default state for untouched requests
Waiting Waiting - for Support Triaged, awaiting agent action
Waiting Waiting - Information needed Asked requester for more details
Closed Closed - Resolved Successfully completed
6

Communicating with the Requester

To send an email to the requester, click the Reply via Email button in the sidebar. This opens the email compose modal.

  • Recipients: Requester's email is suggested — click to add. Type additional emails and press Enter or click "Add".
  • Subject: Pre-filled with the request reference. Edit as needed.
  • Message: Write your response. Sent via the system's SMTP relay.

Click Send Email to dispatch. The email appears in the Activity Stream and is recorded in the Audit Log.

Reply threading: When the requester replies to your email, their response is automatically threaded back into the same ticket via the Outlook Conversation ID. No manual re-linking needed.
7

Closing a Ticket

When a request is fully resolved:

  1. Add a final note explaining what was done (for audit trail purposes)
  2. Click Close Ticket in the sidebar Quick Actions

The status badge changes to a Closed variant and the button switches to Reopen Ticket in case it needs to be revisited.

8

Checking the Audit Log

Navigate to Audit Log in the sidebar to view a comprehensive, immutable record of all actions.
